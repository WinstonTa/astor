# Frontend → Backend Integration Plan

The frontend is fully built but uses hardcoded mock data. We need to wire it to the real backend API. Split into two phases.

---

## Phase 1: Core API Wiring (run flow + live telemetry)

**Goal:** Clicking an agent card starts a real run, and the telemetry log streams live frames from the backend via SSE.

### 1.1 Create API client layer
- **File:** `frontend/src/lib/api.ts`
- Functions: `fetchAgents()`, `startRun(userId, agentId, prompt)`, `confirmRun(runId, decision)`, `fetchCommons(userId)`, `updateCommonsFact(factId, fact)`
- Base URL from env var `VITE_API_URL` (default `http://localhost:3001`)
- Thin fetch wrapper with error handling

### 1.2 Create SSE hook
- **File:** `frontend/src/lib/useRunStream.ts`
- Custom React hook that connects to `GET /api/agent/stream/:runId`
- Parses `ITelemetryFrame` events from the SSE stream
- Returns `{ events: ITelemetryFrame[], status: 'connecting' | 'open' | 'closed' | 'error' }`
- Supports `Last-Event-ID` for reconnection replay
- Auto-closes on unmount

### 1.3 Wire MarketplaceGrid to real agents
- **File:** `frontend/src/components/MarketplaceGrid.tsx`
- Fetch agents from `GET /api/agents` on mount
- Map DB agent records to the existing `Agent` interface (slug → icon mapping, tool_manifest → status)
- Keep mock data as fallback while backend is unavailable

### 1.4 Wire ActiveAgentView to real runs
- **File:** `frontend/src/components/ActiveAgentView.tsx`
- When an agent card is clicked, call `POST /api/agent/run` with `{ userId, agentId, prompt }`
- Store the returned `runId` in state
- Pass `runId` to `useRunStream` hook
- Pass live `events` array to `TelemetryLog`

### 1.5 Update TelemetryLog to render live data
- **File:** `frontend/src/components/TelemetryLog.tsx`
- Accept `events: ITelemetryFrame[]` as a prop (instead of hardcoded ROWS)
- Map each frame to a log row: derive `glyph` and `tone` from `frame.type`
- `thinking` → 💭 phosphor, `tool_start` → 🔧 amber, `viewport_update` → 📸 phosphor, `action_required` → ⚠️ coral, `complete` → ✅ phosphor
- Compute relative timestamps from the first event
- Keep hardcoded ROWS as a demo/preview when no events are provided

### 1.6 Add agent prompt input
- **File:** `frontend/src/components/ActiveAgentView.tsx`
- Add a text input or prompt area so the user can type what they want the agent to do
- Default prompt per agent slug (e.g. "Book a hotel in Seattle under $200")
- The prompt is sent to `POST /api/agent/run`

## Phase 2: Guardrail, Viewport, and Polish

**Goal:** The guardrail modal shows real data from the run, the viewport shows real screenshots, and the Information Commons is editable.

### 2.1 Wire GuardrailModal to real data
- **File:** `frontend/src/components/GuardrailModal.tsx`
- Accept `runId` and `guardrailPayload` as props (from the `action_required` SSE frame)
- Show dynamic hotel name, price, and details from the payload
- On authorize/cancel, call `POST /api/agent/confirm` with `{ runId, decision }`
- Update `ActiveAgentView` guardrail state based on the SSE `complete` or `failed` frame

### 2.2 Update ViewportPanel with live screenshots
- **File:** `frontend/src/components/ViewportPanel.tsx`
- Accept `screenshotUrl` from `viewport_update` SSE frame payload
- When a screenshot URL is available, render it as a real `<img>` instead of the mock hotel cards
- Keep the mock cards as fallback when no screenshot is provided
- The mock browser chrome and scanline overlay remain for visual effect

### 2.3 Add run status awareness
- **File:** `frontend/src/components/ActiveAgentView.tsx`
- Track run status from SSE events (HYDRATING, THINKING, EXECUTING_TOOL, AWAITING_CONFIRMATION, COMPLETE, FAILED)
- Update `StatusRing` dynamically based on run status
- Show different UI states for each phase (loading indicator during THINKING, etc.)
- Handle FAILED state with error message display

### 2.4 Wire Information Commons
- **File:** `frontend/src/components/CommonsView.tsx` (new)
- Fetch user's commons facts from `GET /api/commons?userId=...`
- Display facts in a list with category tags
- Allow inline editing of facts (PUT /api/commons/:factId)
- Triggered from the Sidebar "Information Commons" nav item

### 2.5 Update Sidebar navigation
- **File:** `frontend/src/components/Sidebar.tsx`
- Wire "Information Commons" to render `CommonsView`
- Wire "Security Settings" (placeholder or basic view)
- Pass a `view` state up to `App.tsx` to control which view is shown

## Integration contracts (already frozen)

All frontend changes consume the existing backend API:
- `POST /api/agent/run` → `{ runId }`
- `GET /api/agent/stream/:runId` → SSE `ITelemetryFrame` events
- `POST /api/agent/confirm` → `{ runId, decision }`
- `GET /api/agents` → agent registry
- `GET /api/commons` → Information Commons facts
- `PUT /api/commons/:factId` → edit a fact

No backend changes required for either phase.
